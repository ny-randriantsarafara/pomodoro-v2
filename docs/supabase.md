# Supabase Integration

Rhythm uses Supabase for authentication and cloud-persisted, user-scoped data. Online-only — no offline/local caching.

## Authentication

Three sign-in methods via Supabase Auth:

1. **Magic link** — email-based, passwordless
2. **Google OAuth** — native sign-in
3. **Apple OAuth** — native sign-in

Guest mode is available for users who don't want to create an account.

### Auth interface

```dart
abstract class AuthRepository {
  Future<void> signInWithMagicLink(String email);
  Future<void> signInWithGoogle();
  Future<void> signInWithApple();
  Future<void> signOut();
  Stream<AuthState> onAuthStateChange;
  User? get currentUser;
}
```

### Auth flow

- On app start, check `supabase.auth.currentSession` — valid session routes to `/`, otherwise `/auth`
- `onAuthStateChange` stream drives reactive routing via GoRouter redirect
- Deep link callback: `io.supabase.rhythm://login-callback`

### External configuration (Supabase dashboard)

- Enable Magic Link, Google, and Apple providers
- Configure OAuth credentials (Google Cloud Console, Apple Developer)
- Set redirect URL scheme for deep linking

## Database schema

Three tables, all scoped to the authenticated user via `user_id` + Row Level Security.

### projects

| Column     | Type        | Constraints                                          |
|------------|-------------|------------------------------------------------------|
| id         | uuid        | PRIMARY KEY DEFAULT gen_random_uuid()                |
| user_id    | uuid        | NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE |
| name       | text        | NOT NULL                                             |
| bg_color   | text        | NOT NULL (hex string)                                |
| fg_color   | text        | NOT NULL (hex string)                                |
| created_at | timestamptz | DEFAULT now()                                        |

### tasks

| Column     | Type        | Constraints                                          |
|------------|-------------|------------------------------------------------------|
| id         | uuid        | PRIMARY KEY DEFAULT gen_random_uuid()                |
| user_id    | uuid        | NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE |
| project_id | uuid        | REFERENCES projects(id) ON DELETE SET NULL            |
| title      | text        | NOT NULL                                             |
| completed  | bool        | DEFAULT false                                        |
| created_at | timestamptz | DEFAULT now()                                        |

### sessions

| Column       | Type        | Constraints                                          |
|--------------|-------------|------------------------------------------------------|
| id           | uuid        | PRIMARY KEY DEFAULT gen_random_uuid()                |
| user_id      | uuid        | NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE |
| task_id      | uuid        | REFERENCES tasks(id) ON DELETE SET NULL               |
| task_title   | text        | NOT NULL                                             |
| project_name | text        |                                                      |
| project_bg   | text        |                                                      |
| project_fg   | text        |                                                      |
| preset       | int         | NOT NULL (25, 50, or 90)                             |
| duration     | int         | NOT NULL (actual focused seconds)                    |
| completed_at | timestamptz | NOT NULL                                             |

### Row Level Security

All tables enforce RLS: each user can only SELECT, INSERT, UPDATE, DELETE their own rows (`auth.uid() = user_id`).

### Design decisions

- `sessions` denormalizes `task_title` and project info so history survives task/project deletion
- `ON DELETE SET NULL` on `task_id`/`project_id` keeps orphaned records valid
- IDs are UUIDs generated by Supabase

## Repository layer

All repository interfaces are async (network I/O). Each has a Supabase implementation:

| Interface | Implementation | Supabase table |
|-----------|---------------|----------------|
| `TaskRepository` | `SupabaseTaskRepository` | `tasks` |
| `ProjectRepository` | `SupabaseProjectRepository` | `projects` |
| `SessionRepository` | `SupabaseSessionRepository` | `sessions` |
| `AuthRepository` | `SupabaseAuthRepository` | Supabase Auth |

Models (`Task`, `Project`, `Session`) have `toJson()` / `fromJson()` for Supabase serialization.

`AppStore` (ChangeNotifier) caches data in memory after loading from repos — widget getters stay synchronous while mutations are async.

## App lifecycle

- `main.dart` calls `Supabase.initialize()` with project URL + anon key (via `--dart-define`)
- GoRouter redirect checks auth state for route guarding
- `AppStore.loadData()` fetches tasks/projects/sessions on authenticated home load
- Sign-out triggers `onAuthStateChange`, which redirects to `/auth`

## Edge functions

- `delete-user` — account deletion endpoint (`supabase/functions/delete-user/`)

## Migration

SQL migrations live in `supabase/migrations/`. Deploy with `make deploy-supabase` (or `supabase db push` directly).
